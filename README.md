# OpenAI Gym Reinforcement Learning Lab

## 项目简介

本项目整理了一组围绕 OpenAI Gym 使用习惯构建的强化学习实验，底层实现将使用当前维护中的 Gymnasium 接口与环境生态。仓库目标不是堆砌零散脚本，而是把一系列经典控制、离散决策、连续控制和复杂运动控制任务组织成一套清晰、可运行、可讲解的实验集合。

这些实验覆盖了强化学习中最常见的一条学习路径：先从离散状态和离散动作问题入手，理解价值函数和策略改进；再进入深度强化学习，处理高维状态输入；最后扩展到连续动作控制和更复杂的动力学系统。对于每个实验，仓库都会保留独立目录，用于放置 notebook、结果图、训练记录和补充说明。

## 项目目标

- 系统整理 OpenAI Gym / Gymnasium 常见环境上的代表性强化学习实验
- 用统一的项目结构记录不同算法在不同任务上的建模思路和实现方式
- 让读者不依赖额外背景，也能快速理解每个环境究竟在模拟什么问题
- 为后续补充训练曲线、可视化结果、实验对比和算法分析预留清晰位置

## 实验组织方式

本仓库采用“一个实验一个文件夹”的方式组织内容。每个实验目录后续将至少包含以下内容：

- 一个或多个 `ipynb`：用于展示训练流程、关键代码、结果图和实验说明
- `results/` 或 `artifacts/`：用于保存训练曲线、评估图表和整理后的实验结果

这样组织有两个目的。第一，实验数量增加后，根目录不会被大量 notebook 淹没。第二，实验说明、算法选择、参数依据和结果分析都直接写在 notebook 中，读者打开实验文件夹后就能顺着 notebook 完整阅读，不需要在项目内部再维护大量分散的小型 `README`。

## 方法选择原则

本仓库的方法选择遵循“现代主线优先，教学基线保留，能并行就并行，能使用 GPU 就优先使用 GPU”的原则。不同实验不会机械统一成同一种算法，而是根据环境类型、动作空间、训练成本和展示目标做取舍。

### 1. 现代主线方法优先

如果某个环境适合使用更现代、工程上更稳定、并且更容易发挥设备算力的方法，则优先考虑这类方法。当前仓库里，`PPO` 会作为最核心的主线算法之一，因为它兼顾了以下几点：

- 支持离散动作和连续动作
- 训练过程相对稳定
- 更适合配合并行环境收集轨迹
- 更容易把 `CUDA` 或 `MPS` 利用起来
- 更适合用 notebook 形式做清晰展示

对于适合 `PPO` 的实验，默认优先选择 `PPO`，并在实现上优先考虑并行采样和 GPU 加速。

### 2. 能并行就并行

如果某类算法天然适合多环境并行采样，那么会优先采用并行环境加速训练。这样做的目的不是单纯追求跑分，而是提升训练吞吐、减少等待时间，并让实验结构更符合现代强化学习工程实践。

这意味着像 `PPO` 这类适合批量采样的算法，会优先考虑：

- 多环境并行收集轨迹
- 批量更新策略网络
- 在 `CUDA` 或 `MPS` 上执行神经网络训练

如果某些经典算法本身不适合这种工作流，则不会为了统一形式而强行并行化。

### 3. 教学型方法保留

虽然本仓库整体会偏向更现代的实现路线，但不会因此完全删除经典方法。像 `Q-Learning`、`SARSA`、`DQN` 这类方法仍然保留，原因不是它们一定是所有环境里的最优工程选择，而是它们具有很强的教学价值和方法演化展示价值。

例如：

- `Q-Learning` 适合讲清楚最基础的价值更新思想
- `SARSA` 适合展示 on-policy 和 off-policy 的区别
- `DQN` 适合展示表格方法如何过渡到神经网络近似

因此，像 `DQN` 这类方法在本仓库中的定位，更偏向“经典深度强化学习教学实验”和“方法演进参照系”，而不是所有任务的默认主线方案。

### 4. 按环境类型选算法

本仓库不会把所有环境都统一塞进同一种算法框架，而是按问题结构做区分：

- 小规模离散环境：优先保留 `Value Iteration`、`Q-Learning`、`SARSA`、`Monte Carlo`
- 经典离散深度强化学习环境：可保留 `DQN` 或 `Double DQN`
- 需要现代主线展示的控制任务：优先 `PPO`
- 连续动作环境：优先考虑 `PPO`，必要时辅以 `SAC` 或少量 `DDPG`

### 5. Notebook 中必须写清楚的内容

后续每个实验的 `ipynb` 不只会包含代码，还必须明确写出本实验的建模与实现依据。每个 notebook 至少应说明：

- 为什么选择当前算法，而不是其他常见算法
- 当前环境是离散动作还是连续动作，为什么这会影响方法选择
- 是否启用 `CUDA`、`MPS` 或 `CPU`，为什么这样选
- 是否使用并行环境，为什么使用或为什么不使用
- 关键超参数如何设置
- 学习率、折扣因子、批大小、轨迹长度、更新轮数等参数为什么这样选
- 这些参数改变后通常会带来什么训练行为差异

换句话说，notebook 不只是训练脚本容器，还应承担“实验设计说明书”的角色。读者看到 notebook 时，应该能同时理解环境背景、算法选择、参数依据和结果含义，而不是只看到一段可以运行的代码。

## 基线展示原则

对于适合做直观对比的实验，仓库会优先提供一个“纯随机策略” notebook 作为最基础 baseline。这样做的目的不是把随机策略当成正式方法，而是先让读者建立一个最直接的感受：如果完全不学习、只随机采样动作，环境通常会表现成什么样，任务到底难在哪里。

以 `FrozenLake` 为例，随机策略通常会频繁掉进冰洞，很难稳定到达终点。先展示这个 baseline，再进入 `Q-Learning` 或其他正式方法，读者会更容易看出学习前后的差异，也更容易理解为什么强化学习需要大量交互、探索和价值更新。

## 实验规范

为了让整个仓库保持一致的工程风格、展示风格和阅读体验，后续所有实验默认遵循以下规范。

### 1. 目录组织规范

- 一个实验对应一个独立文件夹
- 一个实验文件夹中可以包含多个 `ipynb`
- 不再默认给每个实验额外维护单独的小型 `README`
- 实验说明、算法解释、参数依据和结果解读优先直接写在 notebook 中

这样做的目的是让读者进入实验目录后，直接顺着 notebook 就能完整理解实验，而不是在多个零散文件之间来回跳转。

### 2. Notebook 组织规范

每个实验的 notebook 不只是代码容器，还应承担实验设计说明书的作用。默认应包含以下内容：

- 这个实验在模拟什么问题
- 为什么选择当前算法
- 为什么不优先选择其他常见算法
- 当前环境是离散动作还是连续动作
- 设备选择依据是什么
- 是否采用并行环境，为什么采用或不采用
- 关键超参数如何设置
- 参数变化通常会带来什么训练行为差异
- 最终结果应该怎么看

如果一个实验有多个 notebook，命名应尽量清晰，例如：

- `random baseline`
- `tabular q-learning`
- `ppo`
- `dqn`

也就是说，实验内部可以存在多个方法对照 notebook，而不是只保留单一路线。

### 3. Baseline 规范

对于适合做直观对比的环境，默认先做一个不学习的 baseline。通常优先考虑：

- 纯随机策略
- 简单启发式策略（如果环境适合）

这个 baseline 的作用不是追求性能，而是让读者先建立环境难度的直观感受，再去理解正式强化学习方法带来的改进。

### 4. 算法选择规范

算法选择遵循“现代主线优先，教学型方法保留”的原则：

- 对现代控制任务，优先考虑 `PPO`
- 对连续控制补充任务，可引入 `SAC` 或少量 `DDPG`
- 对小型离散教学环境，保留 `Value Iteration`、`Q-Learning`、`SARSA`
- 对经典深度强化学习教学环境，可保留 `DQN` 或 `Double DQN`

如果某个环境原生更适合简单方法，不会为了追求“现代”而强行上复杂算法；但如果某个现代方法具有明确的对照价值，也可以额外补一个 notebook 做方法对比。

### 5. 并行与设备规范

- 能并行就并行，尤其是 `PPO` 这类天然适合批量采样的方法
- 能用 `CUDA` 或 `MPS` 时，应在 notebook 中明确写出设备选择依据
- 如果环境或方法本身不适合并行，不要为了统一形式而硬写多进程
- 如果原生向量环境已经足够，就优先用原生方式，不额外引入复杂并发框架

目标是提升训练效率和工程观感，而不是为了“看起来高级”而增加不必要复杂度。

### 6. 运行反馈规范

所有可能运行较久的循环，默认应提供明确进度反馈。通常包括：

- `tqdm` 进度条
- 必要时显示最近 reward、success rate、entropy 等摘要信息

不应让读者在 notebook 运行时长时间面对没有反馈的空白执行状态。

### 7. 结果保存规范

- 训练得到的模型权重、检查点和缓存文件默认不纳入版本控制
- 训练曲线、策略可视化、评估图表、结果摘要表应保留
- 结果文件优先放在实验目录下的 `results/` 或 `artifacts/`
- notebook 执行完成后，输出结果可以直接保存回 `ipynb`，方便打开即读

这样既避免仓库被模型文件污染，也能保留足够完整的实验证据链。

### 8. 可视化规范

实验结果不能只停留在最终一个数字或一张表。对于可视化友好的实验，默认应尽量包含：

- 训练曲线
- 成功率曲线
- 参数变化曲线
- 策略可视化
- 状态价值或动作价值热力图
- 终止原因或回合长度分布图

可视化的目的不是装饰，而是让读者更快理解策略是否真的学到了有效行为。

## 实验总览

### 01. FrozenLake Tabular Q-Learning

这是一个典型的网格世界寻路问题。智能体站在一块结冰的湖面上，需要从起点移动到终点，但地图上分布着多个冰洞，一旦踩空就会立刻失败。表面上看它只是一个简单的走格子任务，实际上它非常适合用来展示强化学习最核心的思想：在不断试错中估计“在某个状态下采取某个动作是否值得”。

这个实验的重点不是复杂建模，而是理解最基础的价值更新机制。智能体每走一步，只能看到当前所在位置和采取动作后的结果，它并不知道哪条路线最好，需要通过大量反复尝试逐步学会绕开危险区域、提高到达终点的概率。这个实验适合作为整个项目的起点，因为它能把“奖励”“探索”“策略改进”“状态价值”这些概念讲得非常直观。

### 02. FrozenLake Value Iteration

这个实验和前一个环境相同，但解决思路不同。这里不再依赖纯试错式学习，而是从动态规划角度出发，直接对每个状态的最优价值进行迭代更新，再反推出应该采取的最优动作。这样可以让读者清楚看到：同一个问题，既可以用基于经验的强化学习方法求解，也可以在已知环境转移结构时用规划方法求解。

这个实验的作用是建立方法之间的参照系。它能帮助读者理解，为什么后面进入深度强化学习时，我们仍然在围绕“状态价值”和“动作价值”做文章，只不过状态空间变大之后，不能再用一张表把所有情况完全列出来。

### 03. CliffWalking SARSA

这是一个非常经典的风险控制实验。智能体需要沿着悬崖边缘从起点走到终点，最短路径往往离悬崖最近，看起来效率最高，但只要一步走错就会掉下去并遭受巨大惩罚。这个环境天然适合说明“激进策略”和“保守策略”之间的取舍。

SARSA 属于 on-policy 方法，它学习的是“当前实际执行策略”下的价值，因此更容易在带有探索的过程中形成相对稳健的行为模式。直观上看，它常常会学出一条稍微绕远、但更安全的路线。这个实验很适合拿来解释：强化学习并不是永远只追求最短路径，很多时候它真正学的是风险和回报的平衡。

### 04. CliffWalking Q-Learning

这个实验仍然是悬崖行走问题，但算法改成 Q-Learning。与 SARSA 相比，Q-Learning 更偏向直接逼近理论上的最优动作价值，因此在很多情况下会更积极地靠近高回报区域。放在这个环境里，它往往更愿意尝试紧贴悬崖的最短路线，因为从理想最优控制角度看，这条路回报最高。

### 特殊说明

#### 19. Acrobot Vertical PPO

`19-acrobot-vertical-ppo` 的目标比原版 `Acrobot` 更强，不再只是越过某个高度，而是要求接近完全竖直并连续保持。这个实验已经记录了多轮尝试与经验总结，但当前结论很明确：仅靠现有这套 `PPO + reward shaping` 设定，还不足以稳定完成目标。

后续如果继续推进这个实验，更合理的方向不是单纯继续堆训练，而是进一步引入运动学、动力学、能量整形或分层控制等更强的结构先验。

这个实验最适合和前一个实验配对阅读。两者面对的是同一张地图、同一套动作、同一组奖励，但行为风格会出现明显差异。它能够非常清楚地说明 on-policy 和 off-policy 的区别，不需要引入高深数学也能看出算法偏好的不同。

### 05. Taxi Tabular Q-Learning

这是一个带有任务流程约束的离散控制问题。智能体扮演出租车司机，需要在有限地图中找到乘客、将其接上车，再送到正确目的地。它不只是“从 A 点走到 B 点”，还包含接客、载客、送达这样一套有顺序的决策链条，因此更接近现实中的流程控制和调度问题。

这个实验的价值在于，它把状态定义和动作效果之间的关系讲得很清楚。出租车所在位置、乘客是否上车、乘客目的地是什么，都会共同影响当前最优决策。智能体既要学会路线选择，也要学会在正确时机执行接客和送客动作，否则即使移动路线正确，任务仍然无法完成。

### 06. Blackjack Monte Carlo Control

这是一个基于纸牌游戏的序贯决策问题。智能体要在不超过点数上限的前提下判断是否继续要牌，目标是在信息不完全、随机性较强的条件下尽量提高胜率。与前面几个环境不同，这里更强调长期统计意义上的收益，而不是单一路径控制。

Monte Carlo Control 的特点是通过完整回合的采样结果来估计状态或动作的好坏，因此这个实验很适合展示“回合结束后再整体回看”的学习方式。它可以帮助读者理解，强化学习并不只适用于机器人和控制系统，也适用于存在随机博弈色彩的决策问题。

### 07. CartPole DQN

这是强化学习里最知名的入门实验之一。一个小车可以在水平轨道上左右移动，小车上连接着一根自由摆动的杆，智能体的任务是在杆即将倾倒前及时推动小车，让杆尽可能长时间保持竖直平衡。这个问题看上去像一个玩具任务，但本质上已经包含了动态系统稳定控制的核心挑战。

难点在于，智能体不能只盯着杆当前是向左还是向右倾斜，它还必须同时考虑小车位置、小车速度、杆角度和角速度，判断下一次施力方向。推得过早、过晚、过强或过弱都可能让系统失稳。这个实验通常是从表格方法迈向深度强化学习的第一站，因为状态已经是连续值，不能再简单靠查表解决。

### 08. CartPole Double DQN

这个实验延续倒立摆小车任务，但算法升级为 Double DQN。它的主要目的不是换一个完全不同的环境，而是在同一个经典控制问题上展示深度价值学习中的一个常见改进方向：减少动作价值的系统性高估。

对于读者来说，这个实验能够说明一个很关键的工程事实：在强化学习里，算法名称变化往往不是“花哨升级”，而是针对训练不稳定、估计偏差、样本效率等问题做的结构性修正。通过对比 DQN 和 Double DQN 的训练曲线与最终策略表现，可以更清楚地理解这类改进存在的必要性。

### 09. Acrobot DQN

这是一个双连杆摆动控制问题。系统由两节杆组成，上端固定，下端自由摆动，智能体不能直接对整个系统任意施力，而是通常只能在关节处施加有限控制，让末端逐步摆到目标高度。这个问题比 CartPole 更难，因为它不是“简单保持平衡”，而是需要先积累动能，再利用摆动把系统甩上去。

这个实验很适合展示“短期动作看起来可能没有立刻收益，但对长期目标有帮助”的强化学习特征。很多控制动作在当前一步看起来并不能马上接近目标，甚至会先往反方向摆动，但长远上却是在为后续冲高做准备。这类行为非常能体现强化学习中的延迟回报特点。

### 10. MountainCar DQN

这是一个典型的稀疏奖励问题。一辆动力不足的小车被困在山谷中，发动机单次输出不够强，无法直接冲上山顶。智能体必须先反方向后退，借助坡度和惯性积累速度，再顺势冲上目标位置。这个问题非常适合说明：最优行为不一定是“朝目标直接前进”。

对于没有接触过强化学习的人来说，这个实验很有解释力，因为它能直观展示“为了最终成功，策略可能先做看似逆目标的动作”。从工程角度看，它也是理解探索策略、奖励设计和训练稳定性的一个重要案例。

### 11. Pendulum DDPG

这是一个经典连续控制实验。一个摆杆固定在支点上，智能体需要在底部或任意初始角度附近，通过连续施加扭矩，把摆杆逐步甩起并稳定在竖直朝上的位置。这里动作不再是“左”或“右”这样的离散选择，而是一个连续数值，表示施加多大的力矩。

这个任务的现实意义很强，因为它对应的就是控制理论中常见的倒立摆摆起与稳定问题。难点在于，智能体既要学会如何把能量注入系统，又要避免在接近直立时用力过猛导致再次失稳。DDPG 适合作为这个实验的第一版方法，因为它专门面向连续动作空间，能够演示 actor-critic 框架如何直接输出连续控制量。

### 12. Pendulum SAC

这个实验仍然围绕同一个摆杆控制任务，但方法切换为 SAC。相比 DDPG，SAC 通常更强调训练稳定性和样本利用效率，也更适合展示最大熵强化学习在连续控制中的优势。

把它放在 Pendulum 上很合适，因为环境足够经典，读者不会被复杂背景分散注意力，可以把重点集中在算法差异本身。这个实验后续很适合配套展示训练曲线、平均回报变化和策略收敛速度，用来说明不同连续控制方法之间的工程取舍。

### 13. MountainCarContinuous DDPG

这是连续动作版本的山地小车问题。与离散版不同，智能体不再只决定“向左推”还是“向右推”，而是要决定推力大小和方向，这使得控制更细粒度，也更接近真实系统中的油门或驱动力调节问题。

这个实验有助于说明，当动作空间从离散变成连续后，问题并不仅仅是“多了几个动作选项”，而是整个策略表示方式都发生了变化。智能体需要学会在不同位置和速度条件下输出恰当的连续控制量，既不能动力不足，也不能无意义地浪费动作幅度。

### 14. MountainCarContinuous SAC

这个实验与前一个环境相同，但改用 SAC。这里的意义在于建立连续控制上的对照组：同一个动力学系统，不同算法会在探索效率、训练稳定性和最终策略平滑性上体现出不同特征。

对读者来说，这类实验能把“算法不是抽象名词”这件事说明白。最终关注点不是谁的缩写更流行，而是谁能更稳定地学会正确的冲坡节奏，谁在训练过程中更少陷入无效探索或震荡策略。

### 15. LunarLander DQN

这是一个非常直观的着陆控制问题。智能体需要控制一个月面着陆器从空中下降，目标是在重力作用下不断修正横向位置、下降速度和机身姿态，尽量平稳地降落到指定着陆区域。离散版本中，策略通常需要在“不开火”“启动主发动机”“使用左侧姿态推进”“使用右侧姿态推进”几类动作之间做选择。

这个环境之所以经典，是因为它把多个控制目标耦合在了一起。主发动机会改变下降速度，侧向姿态推进会影响角度和横向运动，左右支腿接地状态也会影响最终奖励。智能体不能只学会“慢一点落下去”，还必须学会“落在正确位置、保持较好姿态、避免翻倒和硬着陆”。从展示角度看，这是整个仓库里最容易被非专业读者理解的任务之一，因为现实类比非常清晰。

### 16. LunarLander PPO

这个实验继续研究月面着陆问题，但用 PPO 来学习策略。与价值函数主导的 DQN 路线不同，PPO 属于策略优化方法，更适合展示在较复杂控制目标下，策略是如何通过稳定更新逐步改进的。

把 PPO 放在这个任务上有两个好处。第一，它让读者看到同一个“宇宙飞船降落”问题可以有不同算法解法。第二，它有助于后续比较不同方法在着陆平稳性、收敛速度和训练稳定性上的差异。对于项目展示来说，这种横向对比比单独堆环境名更有说服力。

### 17. LunarLanderContinuous SAC

这是月面着陆任务的连续动作版本。与离散版不同，策略不再只决定是否点火，而是需要连续调节主发动机推力和侧向推进强度。这样一来，问题从“有限动作选择”升级成了“细粒度姿态与推力控制”，难度显著增加。

这个实验更接近真实控制系统的表达方式，因为现实中的推进控制通常不是简单开关，而是推力大小和方向的连续调节。智能体需要学会在接近着陆点时更平滑地减速、纠偏和调整姿态，因此它非常适合用来展示连续控制算法在复杂动力学问题上的表现。

### 18. BipedalWalker PPO

这是一个双足行走控制任务。智能体要控制一个二维双足机器人穿过起伏地形，保持身体平衡、协调双腿动作，并持续向前移动。它不像前面的环境那样只控制一个小车或一个摆杆，而是要同时处理多个关节的动作输出，具有更明显的机器人运动控制特征。

这个实验的难点在于多关节协同和长期稳定性。策略需要学会支撑、摆腿、落脚、重心转移这些动作之间的配合关系，任何一个关节控制不当都可能导致机器人失衡摔倒。对于整个项目来说，这个实验可以作为更高复杂度连续控制任务的代表，也能明显提升仓库的整体工程观感。

## 算法覆盖范围

本项目当前规划覆盖以下几类方法：

- 动态规划：Value Iteration
- 表格方法：Tabular Q-Learning、SARSA
- 回合采样方法：Monte Carlo Control
- 深度价值学习：DQN、Double DQN
- 连续控制：DDPG、SAC
- 策略优化：PPO

这些方法的选择基于环境类型和任务特征，而不是单纯追逐新名词。离散动作环境优先采用价值学习或表格方法，连续动作环境优先采用 actor-critic 或策略优化方法。

## 运行规划

本仓库后续将默认围绕 Python notebook 实验流构建，底层主要依赖：

- Gymnasium
- PyTorch
- NumPy
- Matplotlib
- Pandas
- JupyterLab

训练时将优先支持以下几种执行设备：

- `CUDA`：适用于具备 NVIDIA GPU 的环境，优先作为高吞吐训练方案
- `MPS`：适用于 Apple Silicon 设备，使用 PyTorch 的 `mps` 后端进行加速
- `CPU`：作为默认兼容回退方案，用于无 GPU 环境或个别依赖不兼容场景

在 Apple Silicon 设备上，后续实验实现将优先考虑使用 PyTorch 的 `mps` 设备进行训练加速；如果个别实验或依赖在 `mps` 上存在兼容性限制，则保留回退到 CPU 的执行方式。在 Linux/NVIDIA 环境中，则优先使用 `CUDA`。
